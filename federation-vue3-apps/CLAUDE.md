# Karpathy 行为指南

用于减少 LLM 常见编码错误的行为指南。可按需与项目特定指令合并使用。

**取舍：** 这些指南更偏向谨慎而非速度。对于简单任务，请结合实际判断。

## 1. 编码前先思考

**不要想当然，不要掩盖困惑，要显式说明权衡。**

在实现前：
- 明确写出你的假设。如果不确定，就提问。
- 如果存在多种解释，全部说明，不要默默选择其一。
- 如果有更简单的方法，直接指出；必要时提出反对意见。
- 如果有不清楚的地方，先停下，指出困惑点并提问。

## 2. 简单优先

**只写解决问题所需的最少代码，不做猜测性扩展。**

- 不实现超出需求范围的功能。
- 不为一次性代码做抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景添加错误处理。
- 如果你写了 200 行但 50 行就够，就重写精简。

问自己："资深工程师会觉得这段实现过度复杂吗？" 如果会，就继续简化。

## 3. 外科手术式改动

**只改必须改的部分，只清理自己造成的问题。**

编辑现有代码时：
- 不要“顺手优化”相邻代码、注释或格式。
- 不要重构没有问题的内容。
- 保持现有风格，即使你个人有不同偏好。
- 如果发现无关的死代码，可以指出，但不要删除。

当你的改动产生孤立项时：
- 删除由你的改动导致未使用的 import、变量或函数。
- 不要删除既有的死代码，除非被明确要求。

检验标准：每一行改动都应能直接追溯到用户请求。

## 4. 目标驱动执行

**先定义成功标准，再循环迭代直到验证通过。**

把任务转化为可验证目标：
- "添加校验" → "为非法输入编写测试，再让测试通过"
- "修复 bug" → "先写可复现问题的测试，再让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，给出简短计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

强成功标准能让你独立推进循环。弱标准（如“让它能跑”）会导致频繁来回澄清。

---

**这些指南有效的标志：** diff 中不必要改动更少、因过度复杂导致的重写更少、并且澄清问题发生在实现之前而不是出错之后。
